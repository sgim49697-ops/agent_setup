# architectures.md - 실험 대상 아키텍처와 비교 가설

## 공통 원칙

- 모든 아키텍처는 동일한 시나리오, 동일한 모델 예산, 동일한 평가기를 사용한다.
- 아키텍처별 차이는 오직 오케스트레이션, 역할 분리, 검증 루프에서만 만든다.
- 각 아키텍처 폴더는 설명 문서, 설정 파일, 프롬프트 번들, 어댑터 구현을 함께 가진다.

## 아키텍처별 비교 가설

### workflow_pipeline

- 장점: 단계가 고정되어 재현성이 높고, 실험 간 분산이 낮다.
- 약점: 요구가 모호한 과제에서 중간 재해석 능력이 약하다.

### single_loop

- 장점: 설정이 단순하고 탐색/수정 사이클이 빠르다.
- 약점: 종료 조건이 약하면 "완료 선언" 품질이 흔들릴 수 있다.

### scaffolded_single

- 장점: 계획-구현-검증이 명시되어 범용적인 기본형으로 가장 안정적이다.
- 약점: 지나치게 보수적인 검증 루프는 속도를 늦출 수 있다.

### reviewer_optimizer

- 장점: 구현자와 검증자를 분리해 결함 탐지율을 높인다.
- 약점: 추가 호출 비용과 평가 기준 관리 비용이 발생한다.

### manager_workers

- 장점: 대형 과제에서 탐색 병렬화와 역할 분리가 쉽다.
- 약점: 통합 단계에서 컨텍스트 손실과 충돌이 생길 수 있다.

### long_horizon

- 장점: 장기 리팩터링, 체크포인트, 세션 재개에 강하다.
- 약점: 상태 외부화와 resume 계약을 잘 정의하지 않으면 운영 복잡도가 높다.
