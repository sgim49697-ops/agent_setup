# real_eval_rubric.md - 실제품 기준 실평가 루브릭

이 문서는 `benchmark/rubric.md`를 대체하지 않는다. 기존 문서는 하네스 비교를 위한 `lab rubric`으로 유지하고, 이 문서는 실제 사용 가능성과 제품 완성도를 판단하기 위한 `real evaluation rubric`으로 사용한다.

핵심 전제는 명확하다. 디자인은 직접 손으로 세밀하게 만들지 않아도 된다. 필요하다면 Stitch에 높은 권한을 주고, 인터랙티브 디자인과 UI/UX를 강하게 밀어붙여도 된다. 중요한 것은 “누가 만들었는가”가 아니라 “최종 결과물이 실제 사용자 기준에서 충분히 좋은가”다.

단, 시각 polish만 좋다고 통과할 수는 없다. 이 문서는 결과 품질, 반복 안정성, recoverability, 사용자 플로우, 접근성, 반응형 대응까지 함께 본다.

## 1. 적용 대상


- `single_agent`는 현재 defer 상태이므로 기본 실평가 대상에서 제외한다.


## 2. 실평가 전제

- 실제 LLM 연결이 되어 있어야 한다.
- 비교 실험에서는 입력셋, 평가 절차, operator 개입 기준을 통제한다.
- 디자인은 Stitch에 높은 권한을 줄 수 있다.
- Stitch 개입 자체는 감점 사유가 아니다.
- Stitch를 통해 높은 UI/UX 완성도를 끌어올렸다면, 이는 디자인 점수에서 분명히 보상할 수 있다.
- 다만 시각 완성도가 높아도 사용성, 반복 안정성, recoverability가 낮으면 총점은 제한된다.

## 3. 하드 게이트

아래 항목 중 하나라도 실패하면 총점 계산 전에 `실평가 보류` 또는 `실평가 실패`로 판정한다.

- `live_llm_connected = true`
- 핵심 플로우 완주 가능
  - 입력
  - 생성
  - review 또는 final 상태 도달
  - export
- smoke 기준 통과
- 에러 발생 시 recover 가능
- 반복 실행 최소 안정성 충족

## 4. 점수 체계

총점은 100점 만점이다.

| 항목 | 점수 | 평가 의도 |
| --- | ---: | --- |
| 실제 생성 품질 | 16.7 | 생성 결과가 실제로 읽을 만하고 유용한가 |
| 반복 실행 안정성 | 13.3 | 여러 번 돌려도 흐름과 품질이 무너지지 않는가 |
| 사용자 플로우 완성도 | 10.0 | 입력→생성→검토→export 흐름이 자연스러운가 |
| 출력물 품질 | 10.0 | 최종 포스트와 export 산출물이 제품 수준에 가까운가 |
| 복구 가능성 | 6.7 | 실패/에러 이후 사용자가 회복 가능한가 |
| 성능/응답 시간 | 3.3 | 체감상 지나치게 느리거나 막히지 않는가 |
| 접근성/반응형 | 20.0 | 모바일/데스크톱, 키보드, 레이블, 상태 피드백이 강한가 |
| 디자인 완성도와 인터랙션 품질 | 20.0 | 정보 계층, 시각 의도, hover/focus/transition/feedback이 충분히 좋은가 |

### 왜 디자인과 접근성 비중이 큰가

이 실평가는 “구조 실험이 흥미로운가”보다 “실제로 사용 가능한 제품인가”를 더 강하게 본다. 그래서 접근성/반응형과 디자인/인터랙션 품질 비중을 크게 올린다. 특히 Stitch에 디자인 권한을 크게 줄 수 있다는 전제를 두고 있으므로, 최종 UI/UX 완성도가 높다면 이는 분명히 점수로 보상한다.

## 5. Stitch 권한 반영 정책

Stitch는 이 평가에서 숨겨야 할 개입이 아니라, 결과 품질을 끌어올리기 위한 적극적인 수단으로 본다.

### 원칙

- Stitch 개입 자체는 감점 사유가 아니다.
- 높은 권한을 주고 만든 결과라도 사용성이 약하면 고득점 불가다.
- 반대로 Stitch를 적극 활용해 실제로 좋은 인터랙티브 UI/UX를 만들었다면, 디자인 점수에서 명확히 보상한다.

### 디자인 평가에서 함께 보는 요소

- 정보 계층이 명확한가
- 입력/생성/검토/export 흐름이 자연스러운가
- hover / focus / transition / feedback이 살아 있는가
- 모바일 레이아웃이 무너지지 않는가
- 접근성 상태 피드백이 시각적으로도 설계되어 있는가

### 주의

디자인 가산은 아래를 덮어쓰지 못한다.

- 낮은 사용성
- 반복 실행 불안정
- 복구 불가
- 핵심 플로우 미완주

## 6. 실행 프로토콜

실평가는 아래 절차를 기본으로 한다.

1. 동일 입력셋 3개 이상 사용
   - 쉬운 주제 1개
   - 중간 난도 1개
   - 실패 가능성이 있는 까다로운 주제 1개
2. 각 하네스 최소 3회 반복 실행
3. desktop/mobile 모두 확인
4. 최고점이 아니라 중앙값 기준으로 해석
5. 최종 판정은 `pass / hold / fail`과 수치 점수를 함께 남김

## 7. 판정 규칙

### 실평가 통과

- 하드 게이트 통과
- 총점 85 이상
- 반복 실행 성공률 80% 이상

### 실평가 보류

- 하드 게이트는 통과했지만 수동 개입이 많거나 반복 안정성이 낮음
- 혹은 디자인 완성도는 높지만 생성 품질/회복 가능성/실행 안정성이 약함

### 실평가 실패

- 실제 LLM 미연결
- smoke 기준 실패
- 핵심 플로우 미완주
- recover 불가

위 중 하나라도 해당하면 실패로 본다.

## 8. 실평가 기록용 메타데이터 예시

아래 필드는 실제 점수 기록이나 평가 메모에서 함께 남기는 것을 권장한다.

```json
{
  "live_llm_connected": true,
  "stitch_intervention_level": "high",
  "interactive_design_authority": "high",
  "manual_operator_steps": 2,
  "runs_attempted": 3,
  "runs_succeeded": 3,
  "recovery_success_rate": 1.0,
  "real_eval_status": "pass",
  "real_eval_score": 88.4
}
```

필드 의미:

- `live_llm_connected`: 실제 생성이 live LLM에 연결되어 있었는가
- `stitch_intervention_level`: Stitch가 결과 형성에 얼마나 크게 관여했는가
- `interactive_design_authority`: 디자인 권한을 얼마나 넓게 위임했는가
- `manual_operator_steps`: 자동 흐름 외에 사람이 개입한 단계 수
- `runs_attempted`: 전체 시도 횟수
- `runs_succeeded`: 성공적으로 끝난 횟수
- `recovery_success_rate`: 오류 후 정상 회복 성공 비율
- `real_eval_status`: 최종 판정
- `real_eval_score`: 100점 만점 실평가 점수

## 9. 최종 해석 원칙

이 루브릭은 “잘 만들어진 실험”보다 “실제로 다시 돌려도 괜찮은 제품 후보”를 골라내기 위한 기준이다.

따라서 아래 순서로 해석한다.

1. 실제 LLM 연결 여부
2. 핵심 플로우 완주 여부
3. 반복 실행 안정성
4. recoverability
5. 접근성/반응형
6. 디자인 완성도와 인터랙션 품질

즉, 디자인은 매우 중요하지만 최우선 단독 기준은 아니다. 좋은 실평가 결과는 `잘 작동하고`, `여러 번 돌려도 유지되고`, `실패에서 회복 가능하며`, `그 위에 충분히 좋은 UI/UX를 가진 상태`를 뜻한다.
